# Global Claude Code Instructions

## 人格

- サボるな。丁寧に仕事をしろ。曖昧さをなくせ。
- 指示が曖昧な場合は明確になるまで必ず聞け。わかるまで繰り返せ。
- 目的から理解し、指示より良い方法がある場合は提案しろ。
- 常に最善をとれ（実装コスト/リスクはやらない理由にならない）

## Testing Rules

- テスト失敗の修正を依頼された場合、テストをスキップ（`t.Skip()`、`pytest.mark.skip`、`@Disabled`等）したり、テストを削除したりしてはならない
- テストが失敗する根本原因を調査し、プロダクションコードまたはテストコード自体を修正すること
- スキップが本当に正当な場合（環境依存、外部サービス依存等）は、ユーザーに確認してから行うこと
- Go インテグレーションテストは `make itest` で実行する（`go test` 直接ではなく）。Makefile に該当ターゲットがない場合はまず Makefile を確認する

## Go Development

- ユーザーがライブラリ名を言及した場合（例: `my-lib`）、go.mod の依存ライブラリとして解釈する。git ブランチ名とは解釈しない。明示的に「ブランチ」と言われた場合のみブランチとして扱う
- 既存メソッドの削除・大幅変更は、呼び出し元を確認してから行う。ガード条件の追加で済む場合は削除しない

## Harness Rules

- 機能実装時は作業規模に応じたプロセスを適用すること（Heavy: brainstorming+plan+レビュー / Medium: plan+レビュー / Light: 標準フロー）
- 具体的なチェック観点はタスクの性質に応じて coding-golang / testing-golang / reviewing-golang スキルのリファレンスを参照
- Go実装では必要に応じて 実装(coding-golang)→テスト(testing-golang)→レビュー(reviewing-golang) のループを回し、レビューで問題が見つかれば実装に戻って修正する。品質が十分になるまで繰り返すこと
- ユーザーがハーネス改善を要求した場合は、該当するスキルリファレンスを更新し、必要に応じて再現テストスイートにシナリオを追加すること
